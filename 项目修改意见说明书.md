## 二、严重问题（必须修复）

### 🔴 S1. Agent 认证密钥硬编码默认值

**问题描述**：`src/config/index.js:40` 定义了 `pcAgentSecret` 的默认值为 `'agent-dev-secret'`，如果生产环境部署时忘记在环境变量中覆盖此值，任何知道该默认值的外部攻击者都可以调用 Agent API 拉取/操作所有待打印任务。

**影响范围**：`/api/agent/pending-jobs`、`/api/agent/job-complete`、`/api/agent/heartbeat` 三个接口

**风险等级**：**严重** — 直接导致打印数据泄露和任务被恶意操作

**修改建议**：
```javascript
// src/config/index.js
pcAgentSecret: process.env.PC_AGENT_SECRET,  // 去掉默认值
```

并且在服务器启动时检查该值是否存在：
```javascript
// src/index.js 启动时
if (!config.pcAgentSecret) {
  console.error('[FATAL] PC_AGENT_SECRET 未设置，请配置环境变量');
  process.exit(1);
}
```

### 🔴 S3. 0 字节文件上传后创建订单导致服务器崩溃

**问题描述**：上传 0 字节的 PDF 文件可以成功（无文件大小下限校验），但在创建订单时 `price-calc.js` 计算出的 `total_amount` 为 0 或异常值，写入数据库后 `transaction` 返回空结果，`order.routes.js:67` 访问 `rows[0].id` 时报 `TypeError: Cannot read properties of undefined`，**整个进程崩溃**。

**影响范围**：上传 0 字节文件后创建订单 → 500 服务器崩溃

**风险等级**：**高** — 导致服务器进程退出，影响全部在线用户

**修改建议**：
```
修复路径（三处）：

1. src/utils/price-calc.js — file_pages <= 0 时直接返回错误或默认最小价格
2. src/modules/file/file.routes.js — 在 upload handler 中校验文件大小 > 0
3. src/modules/order/order.routes.js — 对 transaction 返回值做防御性检查
```

```javascript
// file.routes.js upload handler
if (!file || fs.statSync(destPath).size === 0) {
  ctx.status = 400;
  ctx.body = { success: false, message: '文件为空' };
  return;
}
```

## 三、重要问题（建议修复）

### 🟠 I1. 订单状态机缺少显式约束

**问题描述**：订单状态（pending_pay → paid → printing → completed → picked）的流转没有任何数据库级别或应用级别的状态机校验，任何 UPDATE 语句直接设置新状态。虽然在代码逻辑中做了前置检查，但并发场景下存在状态跳跃风险（如 paid 和 cancel 同时通过检查）。

**影响范围**：所有订单状态变更接口

**风险等级**：**中-高**（并发场景低概率但后果严重）

**修改建议**：
```sql
-- 方式一：PostgreSQL CHECK 约束（推荐）
ALTER TABLE orders ADD CONSTRAINT valid_status_transition
  CHECK (
    (status = 'pending_pay') OR
    (status = 'paid' AND (OLD.status = 'pending_pay')) OR
    (status = 'printing' AND (OLD.status = 'paid')) OR
    (status = 'completed' AND (OLD.status = 'printing')) OR
    (status = 'cancelled' AND OLD.status IN ('pending_pay', 'paid'))
  );
```

或应用层封装状态机：
```javascript
// src/utils/order-state-machine.js
const TRANSITIONS = {
  'pending_pay': ['paid', 'cancelled'],
  'paid': ['printing', 'cancelled'],
  'printing': ['completed'],
  'completed': ['picked'],
  'picked': [],
  'cancelled': [],
};
```

### 🟠 I2. 打印中 Agent 崩溃导致任务卡死

**问题描述**：Agent 通过 `GET /api/agent/pending-jobs` 拉取任务后，订单状态变为 `printing`。如果 Agent 在此之后崩溃（断网、断电、进程退出），订单会永久卡在 "打印中" 状态，永远不会恢复为 "paid"（重试）或 "completed"。

**影响范围**：PC Agent 打印流程

**风险等级**：**中**— 需要人工介入恢复

**修改建议**：
```javascript
// 方案一：Agent 心跳超时自动回退
// 在 file-cleanup.js 或独立的定时任务中检查
const stuckOrders = await query(
  `UPDATE orders SET status = 'paid', remark = 'Agent超时自动回退'
   WHERE status = 'printing' AND started_at < NOW() - INTERVAL '30 minutes'`
);

// 方案二：Agent 端实现超时重连+断点续传
```

### 🟠 I3. 负数 file_pages 产生负数金额

**问题描述**：`calc-price` 接口不校验 `file_pages` 是否为负数，传入 `-5` 时返回 `total_sides: -5, total_amount: -5`。创建订单时如果用了这个结果，会生成 "倒贴钱" 的订单。

**影响范围**：价格计算和订单创建

**风险等级**：**中**

**修改建议**：
```javascript
// src/utils/price-calc.js calculate 方法开头
if (!file_pages || file_pages <= 0 || !Number.isInteger(file_pages)) {
  throw new Error('页数必须为正整数');
}
```

---

### 🟠 I4. 预览超过 50 页无提示

**问题描述**：`file.service.js` 中 `MAX_PREVIEW_PAGES = 50`，超过 50 页的 PDF 只生成前 50 页预览，但**前端没有任何提示**告知用户只显示了部分页面。用户可能会误认为文件只有 50 页。

**影响范围**：消费者端预览体验

**风险等级**：**中**

**修改建议**：
```javascript
// file.service.js generatePreviews 返回值中增加截断标记
return {
  urls,
  truncated: totalPages > MAX_PREVIEW_PAGES,
  total_pages: totalPages,
  preview_count: Math.min(totalPages, MAX_PREVIEW_PAGES),
};
```

前端在预览区域增加提示横幅：
```html
<div class="alert alert-warning">仅显示前 50 页预览（共 128 页）</div>
```

## 四、一般问题（建议优化）

### 🔵 G1. CORS 配置过于宽松

**问题**：`app.use(cors({ origin: '*' }))` 允许任意域名跨域访问。生产环境应限制为具体域名。

**建议**：
```javascript
const corsWhitelist = process.env.CORS_ORIGIN || 'https://your-domain.com';
app.use(cors({ origin: corsWhitelist.split(',').map(s => s.trim()) }));
```

---

### 🔵 G2. 开发模式泄漏 Stack Trace

**问题**：`src/middleware/error-handler.js` 中 `process.env.NODE_ENV === 'development'` 时返回 `stack` 字段。如果生产环境意外设置为 development，会暴露内部路径和调用栈。

**建议**：增加更严格的判断，或始终删除 stack 字段改用日志记录。

---

### 🔵 G3. file_url 缺少外键约束

**问题**：`orders.file_url` 是 VARCHAR 字段，可以引用不存在的文件路径。测试表明创建订单时使用不存在的 file_url 可以成功。

**建议**：考虑增加文件引用表或在应用层做存在性校验。

---

### 🔵 G4. 0 字节上传缺少下限校验

**问题**：当前只有 `MAX_FILE_SIZE_MB=50` 的上限校验，没有下限。0 字节文件不应该被视为有效上传。

**建议**：在 `file.routes.js` 的 upload handler 中增加文件大小 > 0 的校验。

---

### 🔵 G5. 文件名 XSS 风险

**问题**：`file_name` 字段直接使用 `file.originalFilename` 存入数据库，未经 HTML 转义。前端在订单列表/详情中直接渲染该字段时存在 XSS 风险。

**建议**：
```javascript
// 后端：存储时对文件名做 HTML 转义
const sanitized = file.originalFilename.replace(/[<>&"']/g, '');

// 前端：使用 textContent 而非 innerHTML 渲染文件名
```

---

### 🔵 G6. Redis 队列数据一致性保证

**问题**：打印队列完全存储在 Redis list 中，Redis 重启后队列信息丢失（订单仍为 paid 状态但不在队列中）。虽然不导致数据丢失，但会造成订单阻塞。

**建议**：增加启动时的队列恢复逻辑：
```javascript
// src/index.js 启动时
async function restoreQueue() {
  const { rows } = await query(
    "SELECT id, shop_id, estimated_seconds FROM orders WHERE status = 'paid' ORDER BY paid_at ASC"
  );
  for (const order of rows) {
    await queueEngine.enqueue(order.shop_id, order.id, order.estimated_seconds);
  }
}
```

---

### 🔵 G7. `dev-database.js` SQL 参数拼接风险

**问题**：内存数据库模式的 `dev-database.js` 使用字符串替换拼接触发 SQL，虽然仅用于开发环境，但如果开发过程中使用了包含特殊字符的数据，会导致 SQL 语法错误。

**影响**：仅影响开发环境

**建议**：替换为 pg-mem 支持的参数化查询方式（pg-mem 新版本支持 `$1` 风格参数）。

---

### 🔵 G8. 营业时间硬编码

**问题**：种子数据中营业时间为 `08:00-22:00`，初始化脚本直接写死在 SQL 中。如果商家营业时间不同，需要手动修改数据库。

**建议**：在管理后台提供营业时间配置 UI，种子数据仅作默认值。
